**로컬 우선(local-first)** 아키텍처. 데이터 정본은 로컬, 서버는 선택적 백업. 단계(phase)로 나눠 MVP를 로컬에 집중한다.

---

## 1. 원칙

- **로컬이 정본**: 모든 기능이 로컬 저장소에서 오프라인으로 동작. (ADR-0007)
- **서버는 백업**: 사용 시 전체 데이터를 압축한 단일 아카이브를 S3 객체 하나로 스냅샷 백업/복원. 상시 서버·실시간 동기화 없음.
- **인증은 서버 쓸 때만**: 로컬 MVP는 로그인 없음. (ADR-0008)
- **금융성 데이터 최소 외부 위임**: ADR-0001의 연장선.

---

## 2. 단계 (Phases)

### Phase 1 — 로컬 MVP (인증 없음)
로컬만으로 완결. 대상은 사실상 1인/1기기.

- 로컬 저장소에 가계 데이터 보관(계좌·카드·거래·카테고리·예산·이체·할부).
- import 파이프라인(CSV/엑셀 파싱·정규화·중복·이체 판별)이 **클라이언트에서** 동작.
- 대시보드·거래·예산 등 전 화면 로컬 데이터로 렌더.
- 백업은 선택: 로컬 파일 내보내기(수동) 또는 Phase 2의 S3 백업.

### Phase 2 — 서버(백업·공유)
서버 기능을 켤 때 활성화.

- **인증** 도입(수단은 이 단계에서 확정: 매직링크/클라우드 계정 등).
- **S3 백업/복원**: 전체 데이터셋 압축 아카이브를 S3 객체 1개로 업/다운로드(타임스탬프 버저닝).
- **2인 공유·동기화**: 로컬 우선 유지하며 공유를 얹는 방식은 이 단계에서 설계(스냅샷 병합 vs 레코드 동기화/CRDT). 기존 가계-멤버 데이터 구조는 이를 막지 않도록 유지.

---

## 3. 데이터 흐름

```
[사용자 export 파일]         (은행/카드 CSV·엑셀)
        │ import(클라이언트 파싱·정규화)
        ▼
[로컬 저장소 = 정본] ──(선택, Phase 2)──▶ [S3 단일 압축 아카이브 = 백업]
        ▲                                         │ 복원
        └─────────────────────────────────────────┘
```

---

## 4. 기술 선택 (제안 · 열린 항목)

| 주제 | 제안 | 상태 |
| :-- | :-- | :-- |
| 로컬 저장소 | SQLite(구조·쿼리 유리) 또는 구조화 JSON | 열림 |
| 클라이언트 플랫폼 | local-first엔 네이티브(모바일/데스크톱)가 자연스러움. 웹은 IndexedDB로 가능하나 백업·용량 제약 | 열림 (기존 목업은 웹 데스크톱 시안) |
| 백업 포맷 | 로컬 DB/데이터를 gzip 등으로 압축한 단일 아카이브, 타임스탬프 버저닝 | 열림 |
| 백업 트리거 | 수동 우선, 자동(주기) 후속 | 열림 |
| 암호화 | 로컬은 기기 보안 의존, **백업 아카이브 암호화** 검토 | 열림 |
| 서버 인증(Phase 2) | S3 접근 자격/앱 계정/매직링크 중 택 | 열림 |

---

## 5. 기존 결정에 대한 영향

- ADR-0002(매직링크 P0) → ADR-0008로 부분 대체: 인증은 Phase 2.
- Supabase-Postgres-클라우드 정본 가정 폐기 → 로컬 저장소 + S3 백업.
- NFR-OPS(무료 티어) → S3 최소 사용으로 대체. NFR-PLT(멀티 디바이스 동기화)·FR-HM(공유·초대)·FR-AU(인증)는 **Phase 2**로 재배치.
- 목업의 로그인 화면은 Phase 2용으로 표시.

---

## 6. 열린 항목

- 로컬 저장소 기술과 클라이언트 플랫폼(네이티브 vs 웹) 확정.
- 백업 아카이브 스키마·버전 호환·암호화.
- Phase 2 공유 동기화 방식(스냅샷 vs CRDT).
