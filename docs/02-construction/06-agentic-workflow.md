# 에이전틱 개발 워크플로 (플레이북)

| 항목 | 내용 |
| :-- | :-- |
| 문서 버전 | v0.1 |
| 작성일 | 2026-07-08 |
| 단계 | Construction (meta) |
| 상태 | 🟢 활성 규칙 |
| 근거 | ADR-0009 |

kakebo의 실행·검증 방식. **에이전틱 엔지니어링 + 멀티 에이전트 오케스트레이션 + 검증 중심**을 하나의 루프로 묶는다. AI-DLC(Inception→Construction→Operations) 위에서 동작.

---

## 1. 에이전틱 엔지니어링 루프

한 단위 작업(bolt/feature)마다:

1. **의도(Intent) — 사람**: 무엇을·왜. 요구사항(FR)·ADR로 명시. 없으면 먼저 스펙을 쓴다(SDD).
2. **계획(Plan) — 에이전트**: 스펙을 읽고 접근·영향 파일·태스크로 분해.
3. **실행(Execute) — 에이전트**: 구현 + 테스트 작성.
4. **검증(Verify) — 사람/검증 에이전트**: 인수조건 통과·드리프트 확인(§3). 통과해야 "완료".
5. **기록(Record)**: 커밋(컨벤션 준수), 결정이 생겼으면 ADR.

원칙: 사람은 **의도와 검증**에 집중, 반복 실행은 위임. 스펙이 코드보다 상위(진실원 = `docs/`).

---

## 2. 멀티 에이전트 오케스트레이션

범위가 분리되는 작업은 **역할별 서브에이전트**로 나눠 각자의 좁은 컨텍스트에서 실행하고, **오케스트레이터**(사람 또는 리드 에이전트)가 조율한다. 정의: `.claude/agents/`.

| 역할 | 담당 | 범위 |
| :-- | :-- | :-- |
| `product-owner` | 요구사항·ADR·설계 문서 | `docs/` |
| `ui-ux-designer` | UI/UX 목업·와이어프레임·디자인 토큰 | `designs/` |
| `frontend-engineer` | Expo 앱 구현 | `frontend/`, `designs/` 참조 |
| `backend-engineer` | 백업/복원 서버 (Phase 2) | `backend/` |
| `qa-verifier` | 인수조건 검증·리뷰·드리프트 점검 | 전체(읽기) |

적용 기준(**비례 적용**):
- 작은 변경 → 단일 에이전트로 처리(오케스트레이션 생략).
- 범위가 겹치지 않는 복수 작업 → 병렬 서브에이전트로 팬아웃, 결과를 오케스트레이터가 통합.
- 서브에이전트는 항상 `docs/`(스펙)·`AGENTS.md`(규칙)를 컨텍스트로 읽는다.

---

## 3. 검증·평가 (Verification / Eval-driven)

자율성이 커질수록 검증이 규율의 축. "완료"의 정의를 검증으로 못 박는다.

**인수조건 → 검증 기준**: 각 FR의 인수조건(요구사항·델타 문서)을 테스트/체크로 연결한다. 예: FR-AC-04 "카드대금 인출은 지출에서 제외" → 유닛테스트로 집계 제외 검증.

**드리프트 점검**: 구현이 스펙과 어긋나면(스펙에 없는 동작, 스펙과 다른 규칙) 코드가 아니라 **스펙을 먼저 갱신**하거나 코드를 되돌린다.

**Definition of Done** (모두 충족해야 완료):
- [ ] 관련 FR 인수조건을 만족(테스트 또는 명시적 확인).
- [ ] 스펙↔구현 드리프트 없음(있으면 문서 갱신).
- [ ] 커밋 컨벤션 준수(`Refs:`에 FR/ADR).
- [ ] 새 결정이 있으면 ADR 기록.
- [ ] `qa-verifier`(또는 사람) 리뷰 통과.

---

## 4. AI-DLC와의 매핑

- Inception(비전·요구사항) = 의도 정의.
- Construction(설계·구현) = 계획·실행 + 오케스트레이션.
- 전 단계에 검증 게이트가 걸린다.
