"이 거래의 상대가 **우리 가계의 계좌·카드**인가"를 어떻게 판별하는지 정의한다. 이 판별이 곧 **이체(제외) vs 지출(집계)** 을 가른다. (ADR-0005)

---

## 1. 문제

가져온 거래 한 줄만 보고 "이건 내 다른 계좌로 보낸 돈(이체)"인지 "월세·외부 송금(지출)"인지 알아야 한다. 그런데 한국 금융사 export 파일은 상대 정보가 **제각각**이다.

- 어떤 파일은 상대계좌번호·예금주를 준다.
- 어떤 파일은 적요/내용 한 줄만 준다("카드대금", "홍길동", "월세").
- 상대계좌번호가 **부분 마스킹**될 수 있다(`110-***-4521`).

그래서 단일 규칙이 아니라 **신뢰도 티어**로 판별한다.

---

## 2. 레지스트리 (사용자가 등록하는 것)

내부 판별의 기준점. FR-AC-01/02로 등록.

**Account**
- 계좌번호 → 정규화(숫자만 추출) 후 **해시 저장**(원문 비저장, NFR-PRV).
  - `acct_no_hash` = hash(digits)
  - `acct_no_tail_hash` = hash(마지막 4~6자리) — 부분 마스킹 대응
- 별칭/예금주명(선택): "신한 주거래", 소유 멤버 이름 등 → 적요 텍스트 매칭용.

**Card**
- 발급사명, 카드 별칭, **결제계좌(FK)**, 결제일.
- 카드대금 적요 패턴: 발급사명 + {"카드대금","카드결제","자동납부","(주)…카드"} 등.

> 등록이 매칭 정확도를 좌우한다. 미등록이면 자동 인식률이 낮아지고 수동 보정 비중이 커진다.

---

## 3. 매칭 알고리즘 (우선순위)

가져온 각 행 `t`(출처 `source`, 금액, 날짜, 적요, 있으면 상대계좌/예금주)에 대해 위에서부터 적용, 처음 걸리는 규칙으로 분류.

**R1. 상대계좌번호 완전 일치 → `account_transfer` (신뢰도 高 · 자동)**
- 상대계좌 digits 해시가 등록 계좌(출처 계좌 제외)와 일치.

**R2. 카드 정산 → `card_settlement` (高 · 자동)**
- 출처가 계좌이고, 적요가 등록 카드의 정산 패턴(발급사명/별칭/"카드대금")과 일치, **그리고** 금액이 그 카드의 해당 기간 청구액과 근사(±허용오차) 또는 결제일 부근.

**R3. 상대계좌 부분 일치 + 보강 → `account_transfer` (中 · 제안)**
- 마스킹된 상대계좌의 tail 해시가 등록 계좌와 일치하고, 금액·날짜 또는 예금주명이 보강되면 **후보로 제안**(사용자 확인 필요).

**R4. 이름/별칭/규칙 매칭 → 이체 후보 (中 · 제안)**
- 적요/예금주에 등록 계좌 별칭·소유 멤버명 포함, 또는 사용자가 만든 **이체 규칙**(TransferRule 키워드) 일치.

**R5. 그 외 → `external` = 지출 (기본값)**
- 어떤 내부 신호도 없으면 외부로 보고 **지출로 집계**. (과소집계보다 안전 — 월세가 잘못 빠지지 않음)

### 신뢰도 → 동작

| 티어 | 규칙 | 가져오기 미리보기 동작 |
| :-- | :-- | :-- |
| 高 (auto) | R1, R2 | 자동으로 '이체·제외' 표시, 사용자는 되돌릴 수 있음 |
| 中 (suggest) | R3, R4 | '이체 의심'으로 강조, 확정 전 사용자가 승인/거절 |
| 기본 | R5 | 지출로 처리, 필요 시 수동 이체 전환 |

모든 자동 결과는 **되돌리기 가능**하고, 놓친 건은 수동 수단으로 보정한다(§5).

## 4. 짝 맞추기 (양측 상쇄)

`account_transfer`는 두 계좌를 모두 가져오면 **출금(계좌A)** 과 **입금(계좌B)** 두 줄로 나타난다. 이를 한 쌍으로 묶어(TransferLink) 둘 다 집계에서 빼고 하나의 이동으로 인식한다.

짝 후보 기준:
- 금액이 크기 같고 부호 반대.
- 날짜 ±1일 이내(은행 처리 지연 허용).
- 한쪽의 상대계좌가 다른 쪽 계좌를 가리킴(또는 R3/R4 보강).

`card_settlement`는 입금 측이 계좌가 아니라 **카드**다. 출금(계좌) ↔ 해당 카드의 그 달 이용액 합을 대조해 검증한다(카드 이용내역은 이미 지출로 집계됨, ADR-0004).

**유일성 불변식:** 각 정산·송금은 정확히 한 번만 이체로 잡힌다(TransferLink 유일).

---

## 5. 수동 보정 (자동이 놓칠 때 · FR-AC-04)

- **이체 짝 맞추기**: 출금과 입금(또는 카드)을 사용자가 직접 연결.
- **이체 규칙**: 적요 키워드 → 앞으로 항상 이체(TransferRule). 예: "○○증권", "카드대금".
- **일괄 이체 표시**: 여러 거래 선택 → 한 번에 이체로 전환.
- **개별 전환**: 거래 편집에서 유형을 이체로.

---

## 6. 엣지 케이스

- **상대정보 없는 파일**: 적요 한 줄뿐 → R4/R5만 가능. 규칙·수동 의존.
- **부분 마스킹**: tail 해시 + 금액/이름 보강(R3). 단독 tail 일치만으론 자동 확정하지 않음(오탐 방지).
- **공동 계좌로의 송금**: 공동 계좌도 등록 계좌이므로 내부 이체로 처리(집계 제외). 개인→공동 이동은 지출이 아님.
- **같은 사람 명의 타행 이체**: 등록되어 있으면 내부, 아니면 외부(지출). 등록 유도.
- **부분 결제/선결제 카드대금**: 금액이 청구액과 안 맞아도 카드 정산 패턴이면 `card_settlement` 후보로 제안(中).

---

## 7. MVP 범위

- R1·R2·R5 + 수동 보정 = **MVP 필수**(P0). 자동 高신뢰 + 안전한 기본값 + 수동 안전망.
- R3(부분 매칭)·R4(이름/규칙 자동) = **P1**, 점진 고도화.
- 정규화·해시·tail 매칭 상세는 구현 시 확정. 오차 허용범위(금액·날짜)는 실제 파일로 튜닝.
