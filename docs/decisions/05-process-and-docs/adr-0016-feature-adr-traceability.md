# ADR-0016: 기능(FR)–ADR 추적성 규칙 — 1:1 지향과 파기 규칙

| 항목 | 내용 |
|------|------|
| 상태 | ✅ Accepted |
| 날짜 | 2026-07-16 |
| 단계 | Construction (프로세스) |

## 맥락 (Context)

기능을 만들면 그 결정 근거(ADR)도 함께 남겨, **기능 ↔ 결정 ↔ 히스토리**를 추적 가능하게 유지하려 한다. 이상적으로는 **하나의 기능(FR) = 하나의 ADR**이고, 그 기능의 변화는 해당 ADR의 변경 이력에서 관리한다.

그러나 **엄격한 1:1은 시간이 지나면 반드시 깨진다.**

- **교차관심 결정**: 하나의 결정이 여러 FR에 걸친다. 예) [ADR-0015](../02-finance-logic/adr-0015-money-currency-representation.md) 금액·통화 표현은 FR-TX·FR-DB·FR-AC-04·FR-BG를 동시에 규율한다.
- **기능 진화·분할·병합**: FR이 쪼개지거나 합쳐지고, 결정이 번복된다.
- **FR에 안 붙는 결정**: 아키텍처·인증·기술 스택·프로세스(이 ADR 자신 포함)는 특정 FR의 기능이 아니다.

1:1을 **절대 규칙**으로 강제하면 억지 분할·문서 복제가 생겨 오히려 추적성이 나빠진다. 그래서 1:1은 **지향(default)** 으로 두고, **깨질 때의 처리**를 함께 정의한다.

## 결정 (Decision)

**원칙(지향).** 기능 결정은 FR 단위로 ADR에 남기고(기본 1:1), 그 기능의 히스토리는 해당 ADR의 **변경 이력** 섹션에서 이어 관리한다. 새 이슈·커밋은 이슈 우선(ADR-0014)으로 그 ADR/FR에 연결한다.

**파기 규칙(1:1이 안 맞을 때).**
1. **교차관심 결정** → 하나의 ADR을 **여러 FR에 링크**한다(관련 문서에 FR 목록 명시). FR별로 쪼개 복제하지 않는다.
2. **기능 진화** → 같은 ADR **변경 이력에 이어 적는다**. 결정이 **번복**되면 새 ADR이 옛 ADR을 **대체**(상태 `Superseded`/부분대체)하고 관계를 명시한다(README §새 ADR 규칙 3).
3. **기능 분할/병합** → 원 ADR 유지 + 파생 ADR 추가, **관계도**에 관계를 남긴다.
4. **FR 없는 결정**(아키텍처·프로세스·기술 스택) → 해당 카테고리 폴더에 ADR을 두되 FR 링크는 생략한다.

**불변 최소기준(무엇이 있어도 지킨다).** 모든 기능 결정은 **ADR ↔ FR ↔ 이슈 ↔ 커밋** 사슬에서 끊긴 고리가 없어야 한다. 1:1이 깨지는 방식은 허용하되, **추적 불가능은 허용하지 않는다.**

## 근거

- 1:1은 추적성의 **이상**이지 목적이 아니다. 목적은 "왜 이렇게 했는지"를 언제든 되짚는 것.
- 깨지는 지점을 **미리 규정**해 두면, 깨져도 일관된 방식으로 깨져 추적성이 유지된다.
- 기존 관례(카테고리 폴더·번호=시간순·변경 이력·이슈 우선)와 그대로 합쳐진다.

## 결과 (Consequences)

- (+) 유연하면서도 추적 가능 — 교차관심·진화·번복을 정해진 방식으로 흡수.
- (+) 억지 문서 분할·복제 방지.
- (−) "1개 ADR을 여러 FR에 링크"에는 판단이 든다 → **관련 문서에 FR·관계 명시 의무**로 보완.
- (→) 신규 ADR 추가 시 `README`(카테고리 표·관계도)와 `development-log.md`를 함께 갱신.
