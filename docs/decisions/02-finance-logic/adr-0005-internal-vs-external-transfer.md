# ADR-0005: 내부 이동 vs 외부 지출 구분

| 항목 | 내용 |
|------|------|
| 상태 | ✅ Accepted |
| 날짜 | 2026-07-06 |
| 단계 | Inception → Construction |

## 맥락 (Context)

ADR-0004에 따라 "카드 정산"과 "계좌 간 송금"은 이체로 보고 지출에서 제외한다. 그런데 **모든 송금이 내부 이동은 아니다.**

- 내 계좌 → 내 다른 계좌(또는 우리 카드 결제) = 내부 이동 → 제외해야 함
- 내 계좌 → 집주인(월세), 친구, 외부 수취인 = 외부로 나간 **실제 지출** → 집계해야 함

"송금은 전부 이체"로 뭉뚱그리면 월세 같은 실제 지출까지 빠져 지출이 과소집계된다. 따라서 구분 기준은 "송금이냐 카드결제냐"가 아니라 **상대(수취처)가 우리 가계의 계좌·카드인가**이다.

## 결정 (Decision)

거래의 상대가 **가계에 등록된 계좌·카드**이면 **내부 이동(이체)** 으로 분류해 집계에서 제외한다. 등록되지 않은 외부 수취처로 나간 금액은 **지출**로 집계한다.

이체는 하위 유형을 둔다:

- `card_settlement` — 계좌 → 우리 카드 (카드대금)
- `account_transfer` — 계좌 → 우리 다른 계좌

두 유형은 제외라는 결과는 같지만 **검증 대상이 다르다**: 카드 정산은 해당 카드의 이용액 합과, 계좌 송금은 상대 계좌의 입금 건과 대조한다.

## 근거

- 내부/외부 판별의 핵심 신호는 **수취처가 등록된 우리 계좌/카드인지**이다. 이를 위해 가계 계좌·카드 레지스트리(계좌번호 등)를 유지한다.
- 자동 인식은 (a) 출처(은행 계좌 명세서) + (b) 적요 키워드("카드대금", 카드사명, 상대 계좌명) + (c) 금액·날짜가 등록 카드 청구액/상대 계좌 입금과 일치하는지로 판단한다.
- 자동은 휴리스틱이라 놓칠 수 있어 **수동 보정**을 1급 기능으로 둔다: 이체 짝 맞추기(출금↔입금/카드 연결), 이체 규칙(키워드→항상 이체), 일괄 이체 표시.
- 각 카드 정산·각 송금은 **정확히 한 번만** 이체로 잡혀야 한다(중복 제외 방지).

## 결과 (Consequences)

- (+) 월세·외부 송금 등 실제 지출이 잘못 제외되지 않는다.
- (+) 카드 정산은 이용액과, 계좌 송금은 상대 입금과 대조해 "이 이체가 맞나" 검증 가능.
- (−) 가계 계좌·카드를 사전에 등록해야 자동 인식 정확도가 오른다.
- (→) 미등록/모호 건은 기본을 **지출**로 두고(과소집계보다 안전), 사용자가 이체로 바꾸게 한다.
- (→) 데이터 모델에 counterparty(내부 계좌/카드 FK 또는 외부 라벨)와 TransferLink 필요.
