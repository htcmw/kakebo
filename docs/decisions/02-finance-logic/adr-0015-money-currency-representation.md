# ADR-0015: 금액·통화 표현 — 최소단위 정수 + base_amount + 라운딩 적시

| 항목 | 내용 |
|------|------|
| 상태 | ✅ Accepted |
| 날짜 | 2026-07-16 |
| 단계 | Construction |
| 이슈 | #25 |

## 맥락 (Context)

가계부는 한화(KRW)가 기본이지만 **해외결제(USD 등)** 가 실제로 발생한다. 카드 명세서는 해외 건에 대해 **원거래 통화 금액**과 **원화 청구금액**을 함께 준다(예: $50.00 → 68,000원). 이때 세 가지를 동시에 만족해야 한다.

1. **정밀도** — 금액을 부동소수로 저장하면 `0.1+0.2 ≠ 0.3` 오차가 누적된다.
2. **이중 정보 보존** — "달러로 얼마, 한화로 얼마 나갔는지"를 둘 다 남겨야 한다.
3. **라운딩 투명성** — 외화→한화 환산 시 발생하는 반올림/올림이 **조용히** 처리되면 안 되고, 일어났다는 사실이 데이터에 남아야 한다(감사 가능성).

동시에 집계(대시보드·예산·분담)는 단순해야 하므로, MVP는 환율 환산 로직을 짊어지지 않는다.

## 결정 (Decision)

**1. 최소단위 정수 저장.** 모든 금액은 해당 통화의 **최소단위(minor unit) `integer`** 로 저장한다. KRW=원(소수 0자리), USD=센트(2자리). `REAL`(부동소수) 금지. 소수 자릿수는 컬럼에 저장하지 않고 `currency`(ISO 4217)에서 앱이 파생한다.

**2. 통화 3필드(null 없음).** 거래는 `amount`(원거래 통화 금액) + `currency`(원거래 통화) + `base_amount`(기본통화 청구금액)를 **항상** 채운다. 국내 거래는 `amount == base_amount`, `currency == 기본통화`일 뿐 nullable을 두지 않는다. **집계는 언제나 `base_amount`(기본통화) 하나**로 하여 단일 통화 로직을 유지한다. 해외 판별 = `currency <> household.base_currency`.

**3. 라운딩 적시 — `base_amount_source`.** `base_amount`의 출처를 항상 기록한다.
- `domestic` — 국내(환산 없음), `statement` — 명세서 실제 청구액 그대로(카드사 확정, 우리가 라운딩 안 함), `converted` — 앱이 환율로 계산(**라운딩 발생**을 이 값이 적시).
- `converted` 산출 시 기본통화 최소단위로 **반올림(round half up)** 규칙을 고정한다.

**4. 환율 미저장.** `fx_rate`는 `base_amount / amount`로 파생하며(표시 전용), 이 파생 환율로 금액을 **역산하지 않는다**(두 정수가 진실원).

**5. 저장·ORM.** 로컬 정본(ADR-0007)은 SQLite, 스키마·쿼리는 **Drizzle ORM**(ADR-0011 프론트 스택의 expo-sqlite 연장선). 물리 스키마 상세는 [SQLite 스키마 스펙](../../02-construction/07-sqlite-schema.md).

## 근거

- 최소단위 정수는 통화 계산의 표준 관행이며 정밀도를 구조적으로 보장한다.
- `base_amount` 단일 집계는 이중집계 방지(ADR-0004)와 결이 같다 — 합산 축을 하나로 고정.
- nullable 제거로 소비 측 분기·버그를 없앤다.
- 라운딩을 데이터(`base_amount_source`)로 남기면 "왜 이 한화 금액인가"가 사후에 검증된다.

## 결과 (Consequences)

- (+) 해외결제의 원화·외화가 모두 보존되고, 집계는 단일 통화라 단순.
- (+) 라운딩·환산이 감사 가능.
- (−) 앱에 통화별 소수자리 매핑·표시 포맷터 유틸이 필요.
- (−) 다통화 **환율 환산·합산**은 MVP 미구현(컬럼·구조만 대비) → 필요 시 후속 ADR.
- (→) 계좌번호 해시 등 여타 저장 결정은 import 구현 시 별도로 확정.
