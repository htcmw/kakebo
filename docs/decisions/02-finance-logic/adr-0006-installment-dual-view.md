# ADR-0006: 카드 할부 — 두 관점(구매시점/월청구) 제공

| 항목 | 내용 |
|------|------|
| 상태 | ✅ Accepted |
| 날짜 | 2026-07-06 |
| 단계 | Inception → Construction |

## 맥락 (Context)

한국 카드 할부는 지출 "시점"이 모호하다. 예: 60만원을 3개월 무이자 할부로 결제.

- 카드 **이용내역**엔 보통 구매일에 60만원(할부 3개월 표시)이 **한 번** 찍힌다.
- 월 **대금명세서**엔 매달 20만원씩 3번 나눠 청구된다.

"언제 썼나"로 보면 구매한 달에 60만원이지만, "매달 실제 카드값"으로 보면 20만원씩이다. 어느 쪽도 틀리지 않아 사용자마다 원하는 관점이 다르다.

## 결정 (Decision)

할부 원거래를 **한 건만 저장**하고(구매일·전액 + 할부 스케줄), 지출 집계는 **두 관점을 토글로 제공**한다.

- **구매 시점(purchase)**: 산 달에 전액 집계. 기본값 후보.
- **월 청구(billing)**: 스케줄에 따라 월 청구분만 각 달에 집계.

월청구 관점의 월별 금액은 저장된 할부 스케줄에서 **파생(계산)** 하며, 대금명세서의 월별 할부 청구를 별도 지출 거래로 import하지 않는다(ADR-0004에 따라 정산은 이체로 제외되므로 이중집계되지 않는다).

유이자 할부의 **할부수수료**는 실제 비용이므로 별도 '수수료' 카테고리의 지출로 집계한다.

## 근거

- 원거래 1건 + 스케줄 파생은 데이터 중복 없이 두 관점을 모두 지원한다.
- 대금명세서의 할부 청구를 지출로 넣지 않으므로 이용내역과 겹치지 않는다.
- 무이자 원금은 지출 총액에 영향이 없고(시점만 다름), 수수료만 추가 비용이다.

## 결과 (Consequences)

- (+) "언제 샀나"와 "이번 달 카드값" 두 니즈를 모두 충족.
- (−) 월청구 파생 로직·할부 스케줄 저장이 필요해 구현 비용이 있다.
- (✅) **MVP 범위 확정(2026-07-07): 두 관점 모두 MVP에 포함.** 월청구는 저장된 스케줄에서 파생 계산한다.
- (→) 데이터 모델에 Installment(원거래 FK, months, per_amount, interest, fee) 필요.
