# ADR-0007: 로컬 우선 데이터 관리, 서버는 백업(S3 단일 압축)

| 항목 | 내용 |
|------|------|
| 상태 | ✅ Accepted |
| 날짜 | 2026-07-08 |
| 단계 | Construction |

## 맥락 (Context)

데이터를 어디에 정본으로 둘지 결정한다. 기존 설계는 Supabase(Postgres)를 클라우드 정본으로 가정했으나, 사용자는 다음을 원한다.

- **로컬을 정본**으로 데이터를 관리한다.
- 서버는 "백업하는 느낌"으로만 쓴다.
- AWS를 쓴다면 **S3 하나에 전체 데이터를 압축**해 두는 정도의 최소 서버 사용.
- 로컬만 쓰는 동안은 서버·인증이 필요 없다.

이는 ADR-0001(금융 자동연동 미사용, 로컬 파일 import)과 결이 같다 — 민감한 금융성 데이터를 외부에 위임하지 않는 방향.

## 결정 (Decision)

**데이터 정본은 로컬**이다. 앱은 로컬 저장소에서 모든 기능(import·거래·분류·이체·할부·예산·대시보드)을 수행한다.

서버는 **선택적 백업**이다. 사용 시 전체 데이터셋을 **압축한 단일 아카이브**를 S3 객체 하나로 업로드/복원하는 **스냅샷 백업**만 제공한다. 실시간 서버 DB·레코드 단위 동기화는 MVP 비목표.

2인 공유는 로컬 정본만으로는 불가하므로 **서버(백업/동기화) 단계로 미룬다** (ADR-0008, 아키텍처 phase 참조).

## 근거

- 프라이버시·데이터 소유권: 금융성 데이터를 로컬에 보관.
- 인프라 최소화: 상시 서버·DB 불필요, 무료/저비용 지향(NFR-OPS 갱신).
- 오프라인 동작, 규제·복잡도 감소.

## 결과 (Consequences)

- (+) 서버 비용·복잡도 최소, 오프라인 사용, 즉시 시작(로그인 없음).
- (−) 기기 분실·초기화 시 백업 없으면 데이터 소실 → 백업 유도 필요.
- (−) 두 사람 실시간 합산이 로컬만으로는 불가 → **공유는 Phase 2(서버)** 로 이동. MVP는 사실상 1인/1기기.
- (→) 로컬 저장소 기술(SQLite 등), 백업 아카이브 포맷·복원 UX, 백업 암호화는 아키텍처에서 확정.
- (변경) 기존 Supabase-클라우드-정본 가정 폐기. Postgres/RLS는 Phase 2 서버에 한정.
