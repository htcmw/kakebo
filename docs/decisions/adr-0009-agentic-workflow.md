# ADR-0009: 에이전틱 개발 워크플로 채택 (에이전틱 엔지니어링 · 오케스트레이션 · 검증 중심)

| 항목 | 내용 |
|------|------|
| 상태 | ✅ Accepted |
| 날짜 | 2026-07-08 |
| 단계 | Construction (meta) |

## 맥락 (Context)

2026 상반기 AI 개발 담론은 SDD를 넘어 세 흐름이 함께 간다: **에이전틱 엔지니어링**(사람이 의도 정의·검증, 에이전트가 실행), **멀티 에이전트 오케스트레이션/비동기 백그라운드 에이전트**(개발자가 오케스트레이터), **검증·평가 중심**(자율성이 커질수록 검증이 병목이자 규율의 축). 이 프로젝트는 이 셋을 작업 방식으로 채택한다.

이미 밟은 토대: SDD(`docs/`·ADR), 컨텍스트 엔지니어링(`AGENTS.md`). 그 위에 실행·검증 규율을 얹는다.

## 결정 (Decision)

kakebo의 작업 방식으로 다음 셋을 **모두** 채택한다.

1. **에이전틱 엔지니어링**: 사람이 의도(요구사항/ADR)를 정의하고, 에이전트가 계획→구현→테스트를 실행하며, 사람이 검증한다. AI-DLC 루프와 정렬.
2. **멀티 에이전트 오케스트레이션**: 범위가 분리되는 작업(docs/frontend/backend/검증)은 **역할별 서브에이전트**로 나눠 실행하고, 오케스트레이터(사람 또는 리드 에이전트)가 조율한다. 서브에이전트 정의는 `.claude/agents/`.
3. **검증·평가 중심(verification/eval-driven)**: 모든 기능 요구사항의 **인수조건(FR Acceptance Criteria)** 을 검증 기준(테스트/체크)으로 연결하고, "완료" 전 검증 게이트를 통과한다. 스펙↔코드 드리프트를 감시한다.

운영 상세는 [에이전틱 워크플로](../02-construction/06-agentic-workflow.md).

## 근거

- 자율성↑ → 검증이 실질 병목. 검증을 1급 단계로 두면 드리프트·환각을 조기에 잡는다.
- 범위 분리 + 서브에이전트는 각자의 컨텍스트 창을 좁혀 정확도를 높인다(컨텍스트 엔지니어링의 연장).
- 사람은 의도·검증에 집중, 반복 실행은 위임 → 소규모(부부) 프로젝트의 처리량↑.

## 결과 (Consequences)

- (+) 일관된 실행·검증 규율, 스펙과 코드의 정합 유지.
- (−) 초기 셋업(서브에이전트 정의·검증 기준) 비용. 소규모 변경엔 과할 수 있어 **비례 적용**(작은 건 단일 에이전트).
- (→) 모든 P0 FR에 검증 기준 매핑 필요(요구사항 인수조건 활용). Definition of Done에 검증 포함.
- (→) `.claude/agents/`에 역할 정의, `AGENTS.md`에 워크플로·게이트 명문화.
