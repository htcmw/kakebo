## 0. 재구성 고지 · 동결 (READ FIRST)

**이 문서는 요구사항 v1.0 원본의 부재로 재구성한 baseline이며, 2026-07-22 사용자 검토 완료 후 동결(freeze)되었다.**

- **동결 상태:** 🟢 확정 v1.1 (2026-07-22). 이후 변경은 새 버전/ADR로 처리한다.
- **재구성 배경(이력):** v1.0 확정본은 Google Drive·레포 어디에서도 확보되지 않았다(분실/미접근). 사용자가 "가진 소스로 재구성"을 선택해, 이 문서는 다음 **확보된 소스로만** 구성했다.
  1. **제품 비전서 §6 핵심기능 / §7 Non-Goals** — 기능 골격의 근거.
  2. **[02a 요구사항 델타](02a-requirements-accounts-cards.md)** — 계좌·카드·이체·할부(FR-AC-01~06, FR-TX-04)의 **상세 인수조건이 정본으로 존재** → 그대로 통합.
  3. **레포 FR/NFR ID 인벤토리** — 존재하는 요구사항 식별자.
  4. **ADR 정본**(`docs/decisions/`) — 비전 이후 바뀐 결정.
- **없는 인수조건은 지어내지 않았다(이력).** v1.0에만 있었을 세밀한 인수조건은 부재하므로, 알 수 있는 범위만 인수조건으로 썼다. 남은 세부 기준은 "지어낸 값"이 아니라 **구현 단계에서 확정하는 열린 항목**으로 두며, 다음 표시로 남긴다(동결을 막지 않는다).

  > 🔧 상세 인수조건은 구현 단계에서 확정(해당 FR 구현 이슈)

- **ID 번호는 확정 정본이다.** §2.0의 FR 번호 목록 표가 정본 번호이며, **02a 앵커(FR-AC-01~06, FR-TX-04)를 포함해 그룹별 순차 번호로 확정**했다.
- 본 문서는 기존 델타 [`02a`](02a-requirements-accounts-cards.md)를 **통합(merge)**한다. 02a는 유지 + "통합됨·이력 보존"으로 표시한다(§8).

우선순위: **P0**(로컬 MVP 필수) · **P1**(로컬 MVP 포함, 후순위) · **P2**(Phase 2 / MVP 이후 보류).

## 1. 범위 · Non-Goals

### 1.1 범위 (MVP)

신혼부부(2인)를 위한 가계부. 은행·카드사에서 **직접 내보낸 파일을 가져와**(import) 거래를 정규화하고, 분류·이체 정리·예산·대시보드로 **가계 지출을 한 곳에서** 파악한다. 자동 금융연동은 사용하지 않는다(ADR-0001).

**핵심 방향 (비전 이후 ADR로 재조정):**

- **로컬 우선·무인증 MVP.** 데이터 정본은 로컬(SQLite), 앱을 열면 바로 사용한다(ADR-0007, 0008). 로컬 MVP는 사실상 1인/1기기.
- **2인 공유·서버 백업·인증은 Phase 2**로 이동(ADR-0007, 0008). 비전의 "매직링크 인증"·"두 사용자 실시간 합산"은 로컬 MVP 범위가 아니다.
- **플랫폼:** 모바일 + 웹(반응형). 로컬 SQLite는 네이티브(expo-sqlite)·웹(OPFS) 모두 배선 완료(ADR-0011, 0017).

### 1.2 Non-Goals (비전 §7 유지)

- **금융 자동연동 / 오픈뱅킹 / 마이데이터** — **영구 제외**(ADR-0001).
- 투자·자산관리, 대출·신용 관리 — 제외.
- 영수증 OCR — v2 후보.
- 결제 / 송금 실행 — 제외.
- SMS·푸시 알림 파싱 — v2 보류.

### 2.1 가계 · 멤버 (FR-HM)

**FR-HM-01 (P0·로컬) 가계 · 멤버 로컬 시드**
앱이 최초 실행 시 가계 1개 + 멤버 2명(**지훈, 서연**)을 로컬에 생성한다. 로그인 없음. 이 시드가 있어야 공동/개인 태깅(FR-SH)이 가능하다. (비전 §6-1 재조정, ADR-0007/0008)

- 인수 조건: 최초 실행 시 가계 1개와 멤버 2명(지훈=memberA, 서연=memberB)이 로컬에 자동 생성된다. (디자인 토큰 memberA/memberB와 정합)
- 인수 조건: 로그인·인증 없이 바로 사용한다.
- 인수 조건: 거래·계좌·카드는 이 가계에 귀속되고, 멤버(공동/개인)에 태깅할 수 있다. (§2.6 FR-SH 연동)
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-HM-01 구현 이슈) — 멤버 이름 편집 가능 여부, 시드 재설정 흐름.

**FR-HM-02 (P2·Phase 2) 초대 · 연결 · 2인 공유/동기화**
서버 단계에서 두 사용자가 하나의 가계를 초대/연결로 공유하고 실시간 합산한다. (비전 §6-1)

- **재조정:** 로컬 우선 전환으로 **2인 실시간 공유는 로컬만으로 불가** → 초대/연결·공유는 **Phase 2(서버)**로 이동(ADR-0007, 0008). 로컬 MVP는 단일 가계·단일 기기이며 시드된 2인으로 공동/개인 태깅만 지원한다(FR-HM-01).
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-HM-02 / Phase 2 이슈) — 초대·연결·권한·병합 절차.

### 2.2 계정 인증 (FR-AU)

**FR-AU-01 (P2·Phase 2) 계정 인증**
서버 기능(백업/복원, 향후 공유·동기화)을 켤 때만 인증을 요구한다. (비전 §6-2 재조정)

- **재조정 (중요):** 비전은 인증을 **이메일 매직링크**로 MVP 필수(P0)로 두었다(ADR-0002). 로컬 우선 전환으로 **로컬 MVP는 인증 없음**(앱을 열면 바로 사용)으로 바뀌었다. 매직링크 등 인증 수단은 **Phase 2(서버) 기능**으로 이동한다. → ADR-0002 **부분 대체**, ADR-0007 로컬 정본, ADR-0008 "인증은 서버 사용 시에만".
- 인수 조건(방향): 로컬 전용 사용 중에는 로그인 화면이 나타나지 않는다.
- 인수 조건(방향): 서버 백업/복원을 활성화할 때 비로소 인증한다. **인증 수단은 Phase 2 서버 설계에서 확정**한다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-AU-01 / Phase 2 이슈) — 인증 수단·세션·복구 흐름.

### 2.3 파일 가져오기 (Import) 파이프라인 — 제품의 심장 (FR-IM)

**FR-IM-01 (P0) 파일 가져오기 파이프라인**
사용자가 은행·카드사 웹에서 내보낸 거래내역 파일을 업로드해 정규화·중복제거 후 미리보기로 확정한다. (비전 §6-3, ADR-0001) 제품의 핵심 가치.

비전 §6-3이 명시한 파이프라인 단계(의도):

- 인수 조건(방향): **CSV / 엑셀(.xls, .xlsx)** 업로드를 지원한다.
- 인수 조건(방향): **인코딩 자동감지** — CP949 / EUC-KR / UTF-8을 처리한다.
- 인수 조건(방향): **기관별 매핑 프로파일** — 은행·카드사별 컬럼 양식을 매핑해 표준 거래 형태로 정규화한다.
- 인수 조건(방향): **중복 자동감지·제거** — 중복 키(날짜 + 금액 + 적요 + 출처)로 재업로드·기간 중복 건을 걸러낸다. (ADR-0004 근거의 중복 키)
- 인수 조건(방향): **미리보기 후 확정** — 파싱·정규화 결과를 미리 보여주고, 사용자가 조정한 뒤 확정해야 저장된다.
- 인수 조건(방향): 미리보기에서 **이체로 판정된 건을 구분 표시**하고 확정 전에 조정할 수 있다. (§2.9 FR-AC-04와 연동)
- 인수 조건(금액, ADR-0015): 가져온 금액은 **최소단위 정수**로 저장하고, `amount`·`currency`·`base_amount` 3필드를 항상 채운다. 해외결제는 원거래 통화 금액과 원화 청구금액을 모두 보존한다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-IM-01 구현 이슈) — 인코딩 감지 실패 처리, 지원 기관 목록, 중복 판정 임계·동률 처리, 매핑 프로파일 미지원 양식의 폴백, 대용량/에러 행 처리, 미리보기 편집 범위 등 정량 기준. (구현 상세 일부는 [이체 매칭 규칙](../02-construction/03-transfer-matching.md) 참조)

### 2.4 거래 내역 관리 (FR-TX)

> 앵커: **FR-TX-04(할부)는 02a §3 정본**이다. FR-TX-01~03은 비전 §6-4를 근거로 하며 §2.0 표에서 번호가 확정되었다.

**FR-TX-01 (P0) 거래 목록 · 검색 · 필터**
가져오거나 입력한 거래를 목록으로 보고 검색·필터한다. (비전 §6-4)

- 인수 조건(방향): 거래를 목록으로 조회하고, 기간·카테고리·계좌/카드·공동/개인 등으로 필터·검색할 수 있다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-TX-01 구현 이슈) — 지원 필터 축·정렬·페이지네이션·검색 대상 필드.

**FR-TX-02 (P0) 거래 수동 추가 · 수정 · 삭제**
가져오기로 잡히지 않는 현금 지출 등을 수동으로 추가하고, 잘못 파싱된 건을 수정·삭제한다. (비전 §6-4)

- 인수 조건(방향): 거래를 수동으로 추가·수정·삭제할 수 있다.
- 인수 조건(데이터, [02a §4](02a-requirements-accounts-cards.md)): 거래는 `kind`(income/expense/transfer), `source`(계좌/카드/수동), `counterparty`(내부 계좌·카드 또는 외부 라벨)를 가진다. `transfer`는 category·sharing을 갖지 않는다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-TX-02 구현 이슈) — 필수 입력 필드·검증 규칙·수정 이력.

**FR-TX-03 (P0) 금액 · 통화 표현**
모든 거래 금액은 정밀도·이중정보·라운딩 투명성을 보장하는 형태로 저장·표시한다. (ADR-0015)

- 인수 조건: 금액은 통화 **최소단위 정수**로 저장한다(부동소수 금지). 집계는 언제나 `base_amount`(기본통화) 하나로 한다.
- 인수 조건: 외화→기본통화 환산 시 `base_amount_source`(`domestic`/`statement`/`converted`)로 라운딩 발생을 데이터에 남긴다.

**FR-TX-04 (P1) 카드 할부 처리 — 두 관점** *(출처: [02a §3](02a-requirements-accounts-cards.md) 정본, ADR-0006)*
할부 결제를 원거래 1건 + 할부 스케줄로 관리하고, 지출 집계를 **구매시점 / 월청구** 두 관점으로 전환해 볼 수 있다.

- 인수 조건: 거래에 할부 개월·회차·월 청구액을 지정·표시할 수 있다.
- 인수 조건: '구매 시점' 보기는 구매한 달에 전액, '월 청구' 보기는 월 청구분만 각 달에 집계한다.
- 인수 조건: 대금명세서의 월별 할부 청구는 별도 지출 거래로 적재하지 않는다(이중집계 방지).
- 인수 조건: 유이자 할부수수료는 '수수료' 카테고리의 지출로 집계한다.
- **범위 확정(2026-07-07): 구매시점·월청구 두 관점 모두 MVP에 포함(P1).** 월청구 뷰는 저장된 할부 스케줄에서 파생 계산한다.

### 2.5 카테고리 분류 (FR-CT)

**FR-CT-01 (P0) 카테고리 분류 — 기본 + 사용자정의 + 규칙기반 자동분류**
거래를 카테고리로 분류한다. 기본 카테고리와 사용자정의 카테고리를 두고, 규칙 기반으로 자동 분류한다. (비전 §6-5)

- 인수 조건(방향): 기본 카테고리 세트를 제공하고, 사용자가 카테고리를 추가·수정할 수 있다.
- 인수 조건(방향): 규칙(예: 적요 키워드 → 카테고리)으로 거래를 자동 분류하고, 사용자가 개별 건을 수정할 수 있다.
- 인수 조건(연동, ADR-0006): 유이자 할부수수료는 '수수료' 카테고리로 분류된다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-CT-01 구현 이슈) — 기본 카테고리 목록, 자동분류 규칙 우선순위·충돌 처리, 미분류 기본값.

### 2.6 공동 / 개인 구분 (FR-SH)

**FR-SH-01 (P0) 공동 · 개인 태깅**
각 거래를 **공동 / 개인(지훈 | 서연)** 으로 태깅한다. 무인증 로컬에서 동작하며, 개인 태깅 시 멤버는 로컬 시드된 2인(FR-HM-01) 중 선택한다. (비전 §6-6)

- 인수 조건: 지출 거래를 공동 또는 개인으로 표시하고, 개인일 경우 로컬 시드 멤버(지훈|서연) 중 하나를 지정한다.
- 인수 조건: 로그인·인증 없이 로컬에서 태깅이 동작한다.
- 인수 조건(데이터): `transfer` 유형 거래는 sharing 태깅을 갖지 않는다([02a §4](02a-requirements-accounts-cards.md)).
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-SH-01 구현 이슈) — 기본 태깅값, 태깅 UI.

**FR-SH-02 (P0) 분담 합계**
공동/개인 태깅을 바탕으로 분담 합계를 계산한다. (비전 §6-6)

- 인수 조건(방향): 공동 지출과 각 개인(지훈|서연) 지출의 합계를 산출해 보여준다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-SH-02 구현 이슈) — 분담 비율(50:50 고정 vs 조정 가능), 정산 표시 방식.

### 2.7 대시보드 (FR-DB)

**FR-DB-01 (P0) 대시보드**
이번 달 합산 지출/수입, 카테고리별 요약, 추이를 보여준다. (비전 §6-7)

- 인수 조건(방향): 이번 달 총지출·총수입 합계를 표시한다.
- 인수 조건(방향): 카테고리별 지출 요약을 표시한다.
- 인수 조건(방향): 기간별 추이(월별 등)를 표시한다.
- 인수 조건(집계, ADR-0004·0015): 집계는 `base_amount` 기준이며, **카드 정산·계좌 간 이체는 지출/수입에서 제외**한다(이중집계 방지). 카드 지출은 이용내역에서만 집계한다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-DB-01 구현 이슈) — 추이의 기간 단위·기본 뷰, 위젯 구성, 할부 두 관점(§2.4 FR-TX-04)의 대시보드 반영 방식.

### 2.8 예산 (FR-BG)

**FR-BG-01 (P1) 카테고리별 월 예산 설정 · 실적 비교**
카테고리별로 월 예산을 설정하고 실제 지출과 비교한다. (비전 §6-8)

- 인수 조건(방향): 카테고리별 월 예산 금액을 설정할 수 있다.
- 인수 조건(방향): 설정한 예산 대비 실제 지출(실적)을 비교해 보여준다.
- 인수 조건(집계, ADR-0004·0015): 예산 실적도 `base_amount` 기준·이체 제외 규칙을 따른다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(FR-BG-01 구현 이슈) — 예산 주기(월 고정 여부), 초과 경고·표시, 미설정 카테고리 처리.

### 2.9 계좌 · 카드 · 이체 (FR-AC) — 02a 정본 통합

> 아래 FR-AC-01~06은 [02a](02a-requirements-accounts-cards.md)의 **상세 인수조건을 그대로 통합**한 것이다(정본). 근거: ADR-0004, 0005, 0006. 상세 매칭 규칙은 [이체 매칭 규칙](../02-construction/03-transfer-matching.md).

**FR-AC-01 (P0) 계좌 등록 · 관리**
가계에 은행 계좌를 등록·수정·삭제한다. 계좌는 이름·기관·(마스킹된)계좌번호·소유(멤버 또는 공동)를 가진다.

- 인수 조건: 등록된 계좌 잔액은 가져온 입출금 내역으로 계산·표시된다.
- 인수 조건: 계좌는 특정 멤버 또는 공동으로 소유 지정할 수 있다.

**FR-AC-02 (P0) 카드 등록 · 관리**
가계에 카드를 등록·수정·삭제한다. 카드는 이름·발급사·소유 멤버·**결제계좌**·결제일을 가진다.

- 인수 조건: 각 카드는 하나의 **결제계좌**(등록된 계좌)에 연결된다.
- 인수 조건: 카드의 이번 달 청구 예정액과 결제 예정일을 확인할 수 있다.

**FR-AC-03 (P0) 계좌 · 카드 레지스트리**
등록된 계좌·카드 식별정보(계좌번호 등)를 내부/외부 판별에 사용한다.

- 인수 조건: 거래 상대가 등록된 우리 계좌/카드와 일치하면 내부 이동 후보로 인식한다. (ADR-0005)

**FR-AC-04 (P0) 이체(내부 이동) 처리 — 이중집계 방지**
카드 정산(카드대금)과 계좌 간 송금을 **이체**로 분류해 수입·지출 집계에서 제외한다. (ADR-0004, 0005)

- 인수 조건: 지출은 카드 이용내역·계좌 직접 지출에서만 집계하고, 카드대금 인출은 이체로 제외한다.
- 인수 조건: 상대가 등록된 우리 계좌/카드이면 이체, 외부 수취처이면 지출로 처리한다. (월세·외부 송금이 잘못 제외되지 않는다.)
- 인수 조건: 자동 인식이 놓친 건을 **수동 보정**할 수 있다 — ① 이체 짝 맞추기(출금↔입금/카드 연결) ② 이체 규칙(적요 키워드→항상 이체) ③ 여러 건 일괄 이체 표시.
- 인수 조건: 가져오기 미리보기에서 이체로 판정된 건을 구분 표시하고, 사용자가 확정 전에 조정할 수 있다.
- 인수 조건: 각 정산·송금은 정확히 한 번만 이체로 집계 제외된다.

**FR-AC-05 (P1) 이체 유형 구분**
이체를 `card_settlement`(계좌→우리 카드)와 `account_transfer`(계좌→우리 계좌)로 구분해 검증한다.

- 인수 조건: 카드 정산은 해당 카드의 이용액 합과 대조할 수 있다.

**FR-AC-06 (P0) 이체 자동 인식 규칙**
가져온 거래의 상대가 등록된 우리 계좌/카드인지 판별해 이체/지출을 자동 분류한다. 상세 규칙은 [이체 매칭 규칙](../02-construction/03-transfer-matching.md) 참조.

- 인수 조건: 상대계좌번호가 등록 계좌와 일치하면 이체로 자동 분류한다(신뢰도 高).
- 인수 조건: 계좌 출금 중 등록 카드 정산 패턴+금액 근사 건은 카드 정산 이체로 자동 분류한다.
- 인수 조건: 어떤 내부 신호도 없으면 기본을 지출로 처리한다(외부 송금 오제외 방지).
- 인수 조건: 부분 마스킹 계좌번호·이름/규칙 매칭은 '이체 의심'으로 제안하고 사용자가 확정한다(P1).
- 인수 조건: 계좌번호는 원문이 아니라 정규화 해시로 저장·매칭한다(NFR-PRV).

## 3. 비기능 요구사항 (NFR)

**NFR-PLT-01 (P0) 플랫폼 — 모바일 + 웹 반응형**
앱은 모바일과 웹에서 동작한다. 로컬 SQLite는 네이티브(expo-sqlite)와 웹(OPFS)에서 동일하게 배선되어 있다(ADR-0011, 0017).

- 인수 조건: 로컬 정본 데이터베이스가 네이티브·웹 양쪽에서 동작한다(배선 완료 — ADR-0017 v0.2).
- **참고:** 모바일 반응형 레이아웃 상세는 별도 설계로 분리(#34). 현재 목업은 데스크톱 전용([02a §5](02a-requirements-accounts-cards.md)).
  > 🔧 상세 인수조건은 구현 단계에서 확정(#34 반응형 설계) — 지원 최소 뷰포트·브레이크포인트·브라우저 매트릭스.

**NFR-PRV (P0) 프라이버시 — 계좌번호 원문 비저장 · 해시**
민감 금융 식별정보는 원문으로 저장하지 않는다.

- 인수 조건: 계좌번호는 원문이 아닌 **정규화 해시**로 저장·매칭한다(FR-AC-06 연동). 표시에는 마스킹된 형태를 쓴다.
- 인수 조건(방향): 금융성 데이터는 로컬 정본으로 보관하며 외부에 위임하지 않는다(ADR-0001, 0007).
  > 🔧 상세 인수조건은 구현 단계에서 확정(NFR-PRV 구현 이슈) — 해시 알고리즘·솔트, 로컬 저장 암호화 여부.

**NFR-SEC-02 (P2·Phase 2) 행수준 격리(Row-Level Isolation)**
서버(공유/백업) 단계에서 가계·멤버 간 데이터 격리를 보장한다.

- **재조정:** 로컬 우선 MVP는 단일 기기·무인증이라 행수준 격리 대상이 없다. Postgres/RLS 등은 **Phase 2 서버**에 한정(ADR-0007).
  > 🔧 상세 인수조건은 구현 단계에서 확정(Phase 2 이슈) — 격리 모델 상세.

**NFR-PERF (P1) 성능**
가져오기·목록·집계가 실사용 규모에서 반응성 있게 동작한다.

  > 🔧 상세 인수조건은 구현 단계에서 확정(NFR-PERF 구현 이슈) — 대상 데이터 규모, import/집계 응답시간 목표 등 정량 기준.

**NFR-A11Y (P1) 접근성**
기본적인 접근성을 준수한다.

  > 🔧 상세 인수조건은 구현 단계에서 확정(NFR-A11Y 구현 이슈) — 준거 기준(예: WCAG 레벨), 대상 범위.

**NFR-OPS (P1) 운영 · 비용**
상시 서버·DB 없이 무료/저비용으로 운영한다. 서버는 선택적 스냅샷 백업(S3 단일 압축)만(ADR-0007).

- 인수 조건(방향): 로컬만 사용하는 동안 서버·인증 인프라가 필요 없다.
  > 🔧 상세 인수조건은 구현 단계에서 확정(NFR-OPS / Phase 2 이슈) — 백업 주기·보관·복원 SLA 등.

## 6. 검토 결과 (2026-07-22 확정)

사용자 검토를 완료하고 다음 5개 항목을 확정해 동결했다.

1. **ID 번호 체계 = 재정리·확정.** §2.0 FR 번호 표를 정본으로 확정. 02a 앵커(FR-AC-01~06, FR-TX-04)는 고정, 나머지 그룹도 순차 번호로 확정(FR-HM-01/02, FR-AU-01, FR-IM-01, FR-TX-01~04, FR-CT-01, FR-SH-01/02, FR-DB-01, FR-BG-01). "번호는 재구성이라 다를 수 있다"는 경고는 제거.
2. **세부 인수조건 = '구현 시 확정'으로 동결.** 남은 정량/상세 기준은 각 FR의 `🔧 구현 단계에서 확정` 열린 항목으로 전환. 더는 동결을 막지 않는다.
3. **인증·공유 Phase 2 이동 = 승인.** 비전(매직링크 P0)에서 로컬 무인증 MVP + 인증/공유 Phase 2 재조정(ADR-0002 부분대체, 0007, 0008)을 승인.
4. **로컬 멤버 표현 = 가계+2인 시드, 무로그인.** FR-HM-01(로컬 시드: 가계 1 + 멤버 2명 지훈·서연)과 FR-HM-02(초대·연결·공유 Phase 2)로 분리 확정. FR-SH는 공동/개인(지훈|서연)으로 로컬에서 태깅.
5. **02a 파일 처리 = 유지 + 통합 표시.** 02a는 삭제하지 않고 상단 상태를 "🔵 v1.1로 통합됨·이력 보존"으로 표시(§8).
